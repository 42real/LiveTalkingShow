# 动作素材生成卡住问题记录（2026-06-04）

## 问题现象

在动作素材制作页面点击“直接生成”后，按钮一直显示“生成中”。页面没有报错，也没有明显进度变化，容易误以为前端卡死了。

这次生成的是一段 5 秒左右的绿幕正面人物动作素材，页面参数大致为：`fps=30`、`img_size=256`、`face_det_batch_size=8`、`pads=0,10,0,0`。

## 当时的判断

先看后端进程和端口，`8050`、`8070` 都还在，说明服务不是直接退出。再看动作素材目录，发现目标目录已经创建出来了，并且 `source_clip.mp4` 已经切出来了，但 `full_imgs` 和 `face_imgs` 都还是 0 张。

同时后端日志停在：

```text
Using cuda for face detection.
```

所以可以判断，请求不是卡在浏览器按钮，也不是卡在 FFmpeg 切视频，而是卡在 Wav2Lip 的人脸检测阶段。也就是说，它还没有真正开始把每一帧写成 `full_imgs` 和 `face_imgs`。

## 主要原因

原来的生成方式会对视频里的每一帧都进行人脸检测。`fps` 从 8 改成 30 后，同样长度的视频需要处理的帧数会变多很多。5 秒视频在 30fps 下大约是 150 帧，而 8fps 下只有 40 帧左右。

更重要的是，这次卡点不是单纯慢，而是 CUDA 人脸检测器在批量检测阶段长时间不返回。这样前端请求一直等不到后端响应，就会一直显示“生成中”。

## 处理办法

当时临时把生成逻辑改成：动作素材生成时先检测开始帧的人脸框，然后整段视频复用这个人脸框。

也就是：

```text
首帧检测框 + 同一组 pads -> 得到固定生成框 -> 用这个框处理整段动作视频
```

这样做以后，生成动作素材时不再需要对每一帧都重新跑人脸检测。后端只需要检测一次，然后按照固定框裁剪每一帧的人脸区域。

不过后续测试发现，这个办法会影响嘴型贴回位置。只要人物头部有轻微上下左右移动，固定框就可能让嘴型偏离真实嘴巴。为了嘴部位置更准，当前默认已经调回逐帧人脸检测。

## 以后要遵守的规则

动作素材这种正面人物视频，如果人物头部几乎不动，可以用“首帧检测框 + 同一组 pads”作为应急或者快速测试方案。但这不适合作为默认方案，也不适合嘴型位置要求高的正式素材。

这个规则适合下面这类素材：

- 人物基本正面；
- 头部位置变化不大；
- 人脸没有大幅侧转；
- 手没有明显遮挡脸部；
- 镜头和人物距离比较稳定；
- 绿幕动作素材主要是身体动作变化，而不是镜头运动或者大幅走位。

如果后面素材出现明显侧脸、低头、转身、遮挡、走位很大，或者发现嘴型和嘴巴位置对不准，就应该使用逐帧人脸检测，或者把素材切成更短的片段分别生成。

## 当前代码处理

后端创建动作素材时，当前默认使用逐帧人脸检测。固定人脸框功能仍然保留，但需要显式传 `use_fixed_face_box=true` 才会使用。固定框流程是：

```text
/motion/clips/create
-> 根据 source、start、pads 检测开始帧
-> 得到 padded_box
-> 把 padded_box 作为 fixed_face_box 传给 build_speaking_motion_clip.py
-> build_speaking_motion_clip.py 对整段视频复用这个 fixed_face_box
```

同时增加了半成品清理逻辑：如果上次生成只建了目录，但没有生成 `metadata.json`，下次同名生成前会先清掉这个半成品目录，避免素材库里出现 `frame_count=0` 的无效动作。

## 排查时可以看哪里

如果再次出现“生成中”长时间不动，可以先看这几个地方：

```powershell
netstat -ano | Select-String ':8050|:8070'
```

看服务是否还在。

```powershell
Get-ChildItem -Path "data\speaking_actions\<avatar_id>\<action_id>" -Recurse
```

看是否已经写出 `full_imgs`、`face_imgs`、`coords.pkl` 和 `metadata.json`。

```powershell
Get-Content -Path "logs\backend-wmi.err" -Encoding UTF8 -Tail 120
```

看后端日志停在哪里。如果只看到 `Using cuda for face detection.`，但素材目录里一直没有图片，基本就是人脸检测阶段卡住。

## 备注

`fps=30` 本身仍然会让生成变慢，因为要处理的帧数更多。调参数的时候可以临时用 8 或 15，确认效果后再用 30 生成正式素材。`face_det_batch_size` 主要影响人脸检测批量和显存占用，不是直接提升清晰度的参数。
