# LiveTalking 后端启动失败问题记录

记录时间：2026-06-03 15:03:40 +08:00

## 问题现象

在 Web 前端点击“编排动作”的时候，页面提示：

```text
动作编排失败 {"error":"TypeError: Failed to fetch"}
```

前端服务 `8070` 和 TTS 测试服务 `8036` 都在运行，但 LiveTalking 后端 `8050` 没有稳定监听。实际表现是：有时候 `8050` 会短暂出现，随后又消失，所以前端请求 `/motion/plan` 的时候就连不上后端。

## 当时运行环境

项目目录：

```text
G:\数字人\LiveTalkingShow实验\LiveTalkingShow
```

LiveTalking 使用的 Python：

```text
G:\数字人\livetalking\LiveTalking\.venv\Scripts\python.exe
```

当时三个服务的预期端口是：

```text
8036  RobotTTS 测试服务
8050  LiveTalking 后端
8070  Web 前端
```

## 排查结果

这个问题不是中文路径导致的。LiveTalking 在中文路径下可以正常加载模型并启动，WMI 启动后也已经测通了 `/alpha/session` 和 `/motion/plan`。

真正的问题主要在启动方式上。

第一，使用 PowerShell 的 `Start-Process` 启动后端并重定向日志的时候，当前环境里存在 `Path` 和 `PATH` 重复项，导致 `Start-Process` 报错：

```text
已添加项。字典中的关键字:“Path”所添加的关键字:“PATH”
```

所以这类启动命令没有真正把后端拉起来。

第二，直接在前台执行 `python app.py` 的时候，后端其实可以启动，日志里能看到：

```text
start http server; http://0.0.0.0:8050/index.html
```

但前台运行会一直占着命令。命令一旦超时或者被工具结束，后端进程也会被关闭，所以看起来像是“启动成功了一下，然后又没了”。

第三，使用 `.NET ProcessStartInfo` 方式启动的时候，也能短暂看到 `8050` 监听，但进程仍然可能挂在当前命令下面。当前命令结束后，后端也会跟着消失。

最后用 WMI 创建进程后，后端不会挂在当前工具命令下面，所以能稳定留在后台。

## 已验证的解决办法

推荐用 WMI 启动 LiveTalking 后端。PowerShell 命令如下：

```powershell
$repo = 'G:\数字人\LiveTalkingShow实验\LiveTalkingShow'
$python = 'G:\数字人\livetalking\LiveTalking\.venv\Scripts\python.exe'

Remove-Item -LiteralPath (Join-Path $repo 'logs\backend-wmi.log') -Force -ErrorAction SilentlyContinue
Remove-Item -LiteralPath (Join-Path $repo 'logs\backend-wmi.err') -Force -ErrorAction SilentlyContinue

$commandLine = 'cmd.exe /d /s /c ""' + $python + '" app.py 1>logs\backend-wmi.log 2>logs\backend-wmi.err"'

Invoke-CimMethod `
  -ClassName Win32_Process `
  -MethodName Create `
  -Arguments @{
    CommandLine = $commandLine
    CurrentDirectory = $repo
  }
```

启动后等待一会儿，再检查端口：

```powershell
netstat -ano | Select-String ':8036|:8050|:8070'
```

如果能看到 `8050`，说明 LiveTalking 后端已经在监听。

## 接口验证命令

后端启动后，可以先创建 alpha session：

```powershell
$sessionBody = @{
  avatar = 'mute_teacher_motion_v1_pad01000'
  reuse = $true
} | ConvertTo-Json -Depth 5

$session = Invoke-RestMethod `
  -Method Post `
  -Uri 'http://127.0.0.1:8050/alpha/session' `
  -ContentType 'application/json; charset=utf-8' `
  -Body $sessionBody

$session | ConvertTo-Json -Depth 6
```

再验证动作编排：

```powershell
$sessionId = $session.data.sessionid

$planBody = @{
  sessionid = $sessionId
  avatar_id = 'mute_teacher_motion_v1_pad01000'
  text = '我们先看这个概念的定义。这里是重点，需要特别注意。最后把今天的内容总结一下。'
  max_segments = 4
  use_llm = $true
} | ConvertTo-Json -Depth 8

$plan = Invoke-RestMethod `
  -Method Post `
  -Uri 'http://127.0.0.1:8050/motion/plan' `
  -ContentType 'application/json; charset=utf-8' `
  -Body $planBody

$plan | ConvertTo-Json -Depth 8
```

已验证返回结果里 `provider` 为 `llm`，说明大模型动作编排已经接上。

## 后续注意事项

如果前端再次出现 `TypeError: Failed to fetch`，先不要急着改前端代码。第一步应该先看 `8050` 是否还在监听：

```powershell
netstat -ano | Select-String ':8050'
```

如果 `8050` 不在，优先按上面的 WMI 方式重新启动后端。

中文路径不是这次问题的主要原因，但有些日志会因为编码显示成乱码。后续如果要做给别人部署的版本，可以考虑把正式运行目录放到纯英文路径，减少命令行和老工具的兼容问题。当前本地调试阶段暂时不用搬目录。

本次问题里没有发现 `/motion/plan` 代码注册丢失，也没有发现大模型请求代码本身导致前端失败。前端失败的直接原因是后端 `8050` 没有稳定运行。
